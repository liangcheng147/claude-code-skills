# 笔记编写 Skill

## 定位

技术博客/分享风格的笔记，面向有一定技术背景的读者，兼顾可读性和专业性。

## 工作流程

### 1. 读取材料

根据用户提供的来源类型加载内容：
- **网页链接**：用 WebFetch 抓取内容
- **本地文件**：用 Read 读取
- **代码**：直接读取或用户提供
- **用户粘贴的文本**：直接使用

多个来源时，先全部加载，再按主题整合。

### 2. 分析并列大纲

读完材料后，先理解整体项目结构，再分析当前内容。

**重要性评估方法**：
- 先看整体项目/课程/文档的全局结构，理解各部分之间的关系
- 判断当前章节在整体中的定位：是基础概念、核心方法、还是某个主题的展开？
- 如果某个主题在后续章节会深入讲解，当前只做概述即可
- 如果某个主题是当前材料的核心，且后续不再展开，需要详细处理

示例格式：
```
标题提议：[标题]

大纲：
1. [概念] XXX - 简要说明
   → 无特殊视觉元素
2. [重点] YYY - 核心机制，详细展开
   → 表格：对比不同方案（新建）
   → 图：原资料有 xxx.png，沿用
3. [概念] ZZZ - 背景介绍
   → 无特殊视觉元素
4. [重点] WWW - 实际应用
   → 代码块：关键实现
   → mermaid：工作流程图（新建）
```

大纲中需包含：
- 提议的标题
- 各章节标题及简要说明
- 标注概念/重点
- 标注每个章节需要使用的视觉元素（表格、mermaid、代码块等）

等待用户确认或调整后再动笔。

### 3. 按大纲写正文

根据确认后的大纲逐章节编写。写的时候就遵守风格要求，而不是写完再改。

## 内容深度判断

判断重要性时，必须结合整体项目/课程/文档的全局结构，而不是只看当前章节。

**评估步骤**：
1. 先理解整体结构：这个项目/课程/文档包含哪些部分？
2. 看当前章节与其他章节的关系：某个主题在这里是概述还是深入讲解？
3. 如果某个主题在后续章节会专门展开 → 当前只做概述
4. 如果某个主题是当前材料的核心且后续不再展开 → 详细处理

**概念/引言类 → 简洁**
- 定义、背景、历史、分类介绍
- 在后续章节会专门展开的主题
- 说清楚是什么、为什么需要它即可
- 即使原资料很复杂，也只保留能让读者理解的最小信息量
- 判断标准：如果去掉这部分，读者仍能理解核心内容 → 它就是概念引入

**重点部分 → 详细**
- 核心机制、实现原理、实际用法、常见坑
- 当前材料的核心主题，且后续不再展开
- 深入解释原理、机制、用法
- 使用表格对比、mermaid图示、代码示例等辅助说明
- 判断标准：如果这部分理解不到位，读者无法实际使用这项技术 → 它就是重点
- 如果原资料对重点内容说明不清楚，上网搜索补充，确保笔记内容完整准确

**代码处理**
- 复杂逻辑：逐行或逐块解释
- 简单/直观代码：整体说明作用，代码本身看注释即可
- 代码块前后用一两句话说明上下文

**视觉元素使用原则**

视觉元素的目标是在专业基础上实现图文并茂，提升可读性和美观度。

- **优先使用原资料中的图表**：如果参考资料已有图片、表格、流程图，直接沿用（用相对路径引用）
- **补充新元素**：原资料没有、但确实需要时，再创建新的表格或mermaid图
- **表格**：用于对比、参数列表、配置项、优劣分析等
- **mermaid**：用于流程、架构、关系、状态转换等
- **代码块**：代码示例、配置片段、命令行等
- 大纲阶段就要规划好每个章节用什么视觉元素（标注哪些来自原资料、哪些需要新建）
- 写正文时，视觉元素是内容的一部分，不是装饰
- 每个视觉元素前后需有文字说明：图片前说明场景，图片后解释关键信息；表格前说明对比目的，表格后点出结论

## 写作风格

### 为什么风格重要

笔记的可信度和可读性取决于是否像人写的。模板化、套路化的表达会让读者觉得内容是自动生成的，降低信任感。写的时候就用自然的语气，不需要写完再检查。

### 避免的表达

以下表达会让笔记显得模板化，注意避免：

**模板化强调词**：
"核心概念"、"关键点"、"值得注意的是"、"重要的是"、"需要强调的是"

→ Why：这些词在真人笔记中很少出现，是AI生成内容的典型标志。

**排比/三段式**：
"首先...其次...最后..."、"一方面...另一方面..."、"不仅...而且...还..."

→ Why：真人写作不会刻意追求句式工整，这种结构读起来像演讲稿。

**引导式口语**：
"让我们来看看"、"接下来我们探讨"、"下面介绍一下"、"我们来了解一下"

→ Why：笔记是写给自己或读者看的，不需要"引导"读者进入下一节。

**过度总结**：
每段结尾加"总之"、"综上所述"、"总而言之"，或内容已经很清楚还要再总结一遍。

→ Why：如果内容本身表达清楚，总结是多余的。读者不需要被"提醒"刚才看了什么。

### 正面写法

- 用陈述句直接表达
- 语气平实，像写给自己看的笔记
- 少用形容词堆砌
- 不刻意追求"生动"，清晰即可
- 句式不刻意限制，自然就好

### 示例对比

```
❌ 模板化：
"首先，我们来了解一下GraphQL的核心概念。GraphQL是一种用于API的查询语言，
它具有灵活、高效的特点。值得注意的是，GraphQL与REST有着本质的区别..."

✅ 自然：
"GraphQL是一种API查询语言，客户端可以精确指定需要返回的字段，避免REST中
常见的过度获取或不足获取问题。"
```

## 输出规范

### 语言

跟随源材料语言：中文材料→中文笔记，英文材料→英文笔记。

### 文件命名

只用主题命名，如 `graphql-basics.md`、`react-hooks.md`。

### 输出位置

由用户指定，常见选项：
- 本地 Markdown 文件
- 思源笔记（通过 siyuan MCP）
- 直接在对话中展示
